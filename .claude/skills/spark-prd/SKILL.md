# spark-prd:PRD

把候选 + 商业设计转成可验证的 PRD;MVP 导向,范围克制。

**进入任何 Phase 前,先读目标 idea 的 frontmatter;不满足立即拦截。**

读 `想法池/idea-NNNN_*.md` 的 `artifacts`:

- `artifacts` 不含商业设计文件(`产出/idea-NNNN_商业设计.md`)→ **拦截**,回复:

  **不继续执行任何 Phase。**

- `artifacts` 已含商业设计 → 通过,进 `## 开头必读`。

## 开头必读

激活且通过前置检查后,先读以下文件,再开口:

1. `背景库/_背景索引.md` — 已沉淀的术语与项目背景,避免重复追问已知词。
2. 目标 idea 文件(`想法池/idea-NNNN_*.md`) — frontmatter + body 完整内容。
3. `产出/idea-NNNN_商业设计.md` — **PRD 必须与它的定位/定价/客群保持一致**(这是上游真源)。
4. `调研/idea-NNNN_调研.md`、`决策档案/idea-NNNN_评估_*.md`(最新) — 竞品现实与评估结论,作为需求取舍依据。

读完再开工。

## Phase 0–4

### Phase 0 — 读全部上游

1. 从商业设计提取:定位(价值主张)、目标客群、商业模式、定价分层、GTM、关键风险。
2. 从 idea 提取:`category`、`platform`、`## 完善后描述`、`## 核心假设`。
3. 从调研/评估提取:竞品现实、用户痛点、仍存疑的假设。

**此 Phase 只读文件,不向用户输出内容。**

---

### Phase 1 — PRD 主体

依次产出:

1. **背景与目标**:为什么做、要达成什么(可量化的成功指标 ≥1 条)。
2. **用户与场景(用户故事)**:取自商业设计的目标客群,写成「作为 {角色},我想 {目标},以便 {价值}」若干条。
3. **功能需求**:按优先级分 **P0 / P1 / P2**,**每条必须带验收标准**:
   ```
   [P0] {功能名}:{一句话描述}
        验收:{可观察、可检验的通过条件}
   ```
   P0 = MVP 必须;P1 = 重要非必须;P2 = 以后再说。
4. **非功能需求**:性能/可用性/安全/合规等,只列与本产品真正相关的。
5. **不做的范围(YAGNI)**:明确列出本期**不做**什么,防止范围蔓延。

---

### Phase 2 — 技术方案概要

**轻量、MVP 导向,不过度设计。**

1. **技术选型**:语言/框架/关键依赖/部署形态,一句话说明选它的理由(匹配 platform 与 effort)。
2. **系统结构**:主要组件 + 数据流(用文字或简单列表描述,不强求图)。
3. **关键数据模型**:核心实体 + 关键字段(只列 MVP 必需的)。
4. **主要技术风险**:2–3 条,每条给一句缓解思路。

---

### Phase 3 — 里程碑与任务清单

1. **里程碑**:把到第一个可用版本(MVP)切成 2–4 个里程碑,每个有明确产出。
2. **任务清单**:可执行的 task list,按里程碑归组,每条是一个能独立完成的小任务(承接 PRD → 架构 → 任务拆分的习惯)。颗粒度参照「一个任务一次能做完」。

---

### Phase 4 — 写盘

**4a. 写 PRD 文档**

写入 `产出/idea-NNNN_PRD.md`(结构见下方「PRD 文档结构」)。

**4b. 回填 idea frontmatter**

严格按 `规则/schema与阶段定义.md` 字段名:

```yaml
artifacts:
  - 产出/idea-NNNN_商业设计.md   # 原有,保留
  - 产出/idea-NNNN_PRD.md        # 追加
status: documented               # 终态:已出设计文档
updated: {今天日期}
```

**4c. 更新状态总览**

在 `想法池/_状态总览.md` 对应行更新 `status` 为 `documented`、`updated` 为今天日期。

**4d. 回执**

```
PRD 完成:idea-NNNN「{标题}」→ documented。
  功能需求:P0 {a} 条 / P1 {b} 条 / P2 {c} 条(均带验收标准)
  技术方案:{选型一句话}
  里程碑:{N} 个,任务 {M} 条
  产出:产出/idea-NNNN_PRD.md
这个想法已走完想法引擎全流程(捕捉→完善→调研→评估→商业→PRD)。
```

## PRD 文档结构

写入 `产出/idea-NNNN_PRD.md`:

```markdown
---
idea: idea-NNNN
date: YYYY-MM-DD
based_on: 产出/idea-NNNN_商业设计.md
---

# idea-NNNN PRD

## 背景与目标

## 用户与场景(用户故事)

## 功能需求(P0 / P1 / P2,每条含验收标准)

## 非功能需求

## 不做的范围(YAGNI)

## 技术方案概要(选型 / 结构 / 数据模型 / 风险)

## 里程碑与任务清单
```

## 一致性与克制

- **与商业设计一致**:定位、定价、目标客群必须与 `产出/idea-NNNN_商业设计.md` 一致;若发现冲突,**以商业设计为准**,并在 PRD 里标注「与商业设计 X 处冲突,已按商业设计取值」。
- **需求可验证**:每条功能需求必须有可观察、可检验的验收标准;写不出验收标准的需求说明它还没想清楚,先标 P2 或移入「不做的范围」。
- **范围克制**:先 MVP。把"锦上添花"的功能放 P2 或明确不做,don't gold-plate。
- **证据纪律**:涉及竞品/市场的判断只引调研 `[已核实]`;推断标 `[AI推测]`,不脑补。
