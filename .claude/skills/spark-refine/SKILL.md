# spark-refine:完善

**职责:把 raw 一句话养成可调研的 refined 想法;不调研、不评估。**

---

## 开头必读

激活后先读,再开口:

1. `背景库/_背景索引.md` — 已沉淀的术语与项目背景,避免重复追问已知词。
2. 目标 idea 文件(`想法池/idea-NNNN_*.md`) — 读取 raw 原文与当前 frontmatter。
3. `规则/品类与平台映射.md` — Phase 3 自动填 platform 时参照此表。

读完再说话。

---

## Phase 0–7(核心流程)

> 纪律:**一次只推进一个 Phase,问完一个问题后停下,等用户回应,再推进。**

### Phase 0 — 读原子 + 预热

- 读 raw 原子全文(frontmatter + body `## 原文 [本人]`)。
- **第一个写盘动作**:把 `status` 由 `raw` 改为 `refining`(标记本想法进入完善中;若会话中途断了,下次能认出它在推进中,而非未碰过的 raw)。
- 扫 `想法池/` 所有原子的 `tags`、`category`、`death_reason`(有则读)——为 Phase 7 收尾找火花预热。
- **不向用户开口**,内部准备好再进 Phase 1。

### Phase 1 — 复述 + 破冰问题

- 用**一句话**复述用户的原始想法(不改写,只确认理解)。
- 紧接问**第一个问题**:

  > 「这解决的是谁的什么痛?能说具体点吗?」

- 停下,等用户回应。

### Phase 2 — 语言陷阱检测

- 审查用户回应(及 raw 原文)中的核心词。
- **若发现模糊未定义的关键词**(如"高效""高级""智能""更好""方便""颠覆"):
  - 逼用户定义,举例:「你说的"高效"是省时间还是省钱?或者是减少出错?」
  - **一次只问一个词**,定义不了 → 触发降级流程(见下方「降级判据」)。
- 若核心词清晰 → 直接进 Phase 3。

### Phase 3 — 产品形态 + category/platform

- 根据对话内容,判断产品形态(App / SaaS / 游戏 / 内容 / 工具 / 硬件 / …)。
- 按 `规则/品类与平台映射.md` **自动填** `category` → `platform`。
- 向用户确认或允许覆盖:

  > 「我把这个定成 {category}·{platform},对吗?还是你有别的想法?」

- 停下,等用户确认或给出覆盖值。

### Phase 4 — 核心假设

- 和用户一起抽 1–3 条**核心假设**:这个想法要成立,依赖什么前提?
- 格式引导:「这想法成立,依赖几个前提——你觉得哪条最关键?」
- **一次只提一条假设**:提出后停下等用户确认/修正,确认一条就写进 `## 核心假设` 一行,再提下一条。不要一次抛出 1–3 条让用户一起回。

  ```
  - [假设] {假设内容}
  ```

- 攒够 1–3 条且用户认可后,再进 Phase 5。

### Phase 5 — 主观打分 + 工作量初估

- 问 excitement:

  > 「1–5 分,你对这个想法有多兴奋?」(这是 `[本人]` 数据,只有你知道)

- 等用户回答后,AI 给出 effort 初估(S/M/L/XL,标 `[AI推测]`),并说明理由(一句话)。
- 停下,等用户对 effort 给出反馈或接受。

### Phase 6 — 列待调研清单

- 基于 category + platform + 核心假设,列出:
  - **竞品关键词**(≥2 个,用于 `/调研` 时搜索)
  - **要核实的事实**(≥1 条,如定价区间、用户规模、技术可行性)
- 格式写进 `## 待调研`:

  ```
  - 竞品关键词:{关键词1}、{关键词2}
  - 要核实的事实:{事项1}
  ```

- 向用户确认或让其补充遗漏的关键词。

### Phase 7 — 收尾找火花

- **扫池找火花**:遍历 `想法池/` 所有原子(含 no-go 原子),看 tags / category / death_reason 与当前想法是否形成互补或组合。
  - 若有匹配 → 提示:「{idea-NNNN}(标题)可能和这个互补,可以用 `/找火花` 深挖。」
  - 若无 → 不强行找。
- **写盘**(见下方「收尾写盘」)。

---

## 出闸条件

**四条全部满足,才把 `status` 改为 `refined`;缺一不改。**

① 能一句话说清「为谁解决什么痛」
② `category` + `platform` 已确定
③ `## 核心假设` ≥ 1 条
④ 竞品关键词 ≥ 2 个(写进 `## 待调研`)

不满足 → **不得**将 status 标为 refined,保持 refining 并注明缺哪一条。

---

## 降级判据

以下情况**不进调研,留池当燃料**:

| 判据 | 打 tag |
|---|---|
| 核心词经追问仍无法定义(如"更好""智能"说不清楚) | `#伪概念` |
| 说不出谁用 / 解决什么具体痛 | `#太空泛` |
| 本质是情绪/愿望,非产品形态(如"让世界更美好") | `#不是产品` |

**降级操作:**
- `status` 回滚为 `raw`(Phase 0 已设成 refining,硬降级把它退回池底当 raw 燃料)。
- 在 `tags` 里加对应 tag。
- 在 `## 完善记录 [AI 元记录]` 里写明降级原因(一句话)。
- **绝不删除原子**——死原子是找火花的燃料。

> 区分两种「没出闸」:① **降级**(伪概念/太空泛/不是产品)→ 回滚 `raw` + 打 tag;② **只是还没问完**(信息不全、会话暂停)→ 保持 `refining`,下次接着问。

告知用户:「这个想法现在还太模糊/不够具体,我把它留在池里打上 `#太空泛`,等你想清楚了再回来完善。」

## 背景钩子

遇到以下情况,**暂停完善流程,提示走 `/背景`**:

- body 里发现 `待补背景:[词]`(捕捉时留下的标记)。
- 对话中出现不确定术语或私有项目引用,AI 无法确认含义。

提示方式:「'{词}' 我不太确定,要先用 `/背景` 查清楚吗?查完再继续完善。」
等用户确认后再跳转,不擅自联网查。

## 收尾写盘

Phase 7 完成、出闸条件全满后,更新以下内容:

### frontmatter 更新字段

严格按 `规则/schema与阶段定义.md` 的字段名(frontmatter 里只写纯值,**不要把来源标签写进 YAML**——那不是合法 YAML):

```yaml
status: refined
category: {确认值}
platform: {确认值}
effort: {S/M/L/XL}
excitement: {1-5}
tags: [...已有 + 本次新增]
updated: {今天日期}
```

来源标签写在正文:`effort` 是 AI 估的、`excitement` 是用户给的——如需留痕,在 `## 完善记录 [AI 元记录]` 写一句(如「effort=M [AI推测];excitement=4 [本人]」)。

### body 填写

- `## 完善后描述`:一段话写清「谁的什么痛 + 初步产品形态」。
- `## 核心假设`:逐条写 `- [假设] {内容}`。
- `## 待调研`:竞品关键词 + 要核实的事实。
- `## 完善记录 [AI 元记录]`:记关键转折、用户否决的方向、被丢弃的产品形态、降级原因(若适用)。

### 状态总览更新

在 `想法池/_状态总览.md` 对应行更新:status / category / platform / excitement / effort / updated。

### 收尾回执

```
完善完成:idea-NNNN「{标题}」→ refined。
核心假设 {N} 条,竞品关键词:{关键词}。
下一步:/调研 idea-NNNN
```

## 对话纪律

- **一次只问一个问题,问完停下等用户回应。**
- 不要一口气把 Phase 0–7 全跑完再输出。
- 每个 Phase 有且只有一个向用户提出的问题(或确认)。
- 若用户一次回答了多个问题 → 依次确认,不跳跃。
- 完善只养想法:**不联网调研、不下可行性结论**(那是 `/调研`、`/评估` 的事)。

中文进中文出。用户用英文输入 → 对话用英文,但 body 各节标题保持中文。
