# spark-research:调研

**职责:联网扒真实竞品,每条带来源;不养想法、不下可行性结论。**

---

## 开头必读

激活后先读,再开口:

1. `背景库/_背景索引.md` — 已沉淀的术语与项目背景,避免重复追问已知词。
2. 目标 idea 文件(`想法池/idea-NNNN_*.md`) — 读取 frontmatter + `## 待调研` 节。
3. `规则/品类与平台映射.md` — 决定本次调研主源。

读完再开工。

---

## 前置检查

**进入任何 Phase 前,先核查以下两条;任一不满足立即拦截。**

### 检查 1:status 须 ≥ refined

读目标 idea 的 `status` 字段:

- `status` 为 `raw` 或 `refining` → **拦截**,回复:
  > 「这个想法还没完善,`## 待调研` 竞品关键词缺失。请先 `/完善 idea-NNNN`,把竞品关键词理出来再调研。」
  **不继续执行任何 Phase。**

- `status` 为 `refined`(或更后) → 通过,继续。

### 检查 2:## 待调研 须有竞品关键词

读 idea body 的 `## 待调研` 节,确认有 ≥ 2 个竞品关键词:

- 关键词缺失或不足 → **拦截**,同上提示先 `/完善`。
- 关键词足够 → 通过,读 `规则/品类与平台映射.md` 确定调研主源,进 Phase 0。

---

## Phase 0–5

### Phase 0 — 读 idea + 确定调研主源

1. 从 `## 待调研` 提取:
   - 竞品关键词列表
   - 要核实的事实列表
2. 从 frontmatter 读取 `category` 和 `platform`。
3. 按 `规则/品类与平台映射.md` 的「调研主源」列确定本次主源:
   - 游戏 / Steam → Steam 同类标签榜、SteamDB
   - App / iOS → App Store 榜单、类似 app
   - SaaS / Web → 通搜 + Product Hunt
   - 工具 / Web → 通搜 + Product Hunt + GitHub
   - 内容 / 抖音·公众号 → 平台搜索 + 对标账号
   - 硬件 / 众筹 → 京东众筹 / 嘉立创 + 电商榜
4. 向用户简述:「正在调研 idea-NNNN,主源:{主源},关键词:{词}。开始搜索……」

**此 Phase 只读文件 + 确定策略,不联网。**

### Phase 1 — 竞品发现(web_search)

对每个竞品关键词,用 `web_search` 搜索 **关键词 + 主源平台**,目标是列出候选竞品名单:

- 搜索策略示例:
  - Steam 游戏 → `"{关键词}" site:store.steampowered.com` 或 `SteamDB {关键词} similar games`
  - App Store → `"{关键词}" app store top charts`
  - SaaS/工具 → `"{关键词}" site:producthunt.com` 或 `best {关键词} tools`
  - 内容 → `"{关键词}" 抖音账号` 或 `"{关键词}" 公众号 排行`
- 从结果中识别出 **候选竞品名单**(产品名 + 一句话描述)。
- 若第一轮无结果 → 换关键词同义词或去掉限定词再搜一轮。

**记录每条搜索来源 URL,供 Phase 3 打标签。**

### Phase 2 — 竞品深挖(web_reader)

对 Phase 1 列出的每个候选竞品,用 `web_reader` 拉取官网/商店页/Product Hunt 页,提取:

| 维度 | 内容 |
|---|---|
| 做什么 | 一句话核心功能 |
| 定价 | 价格区间(免费/订阅/买断/内购),标日期 |
| 体量 | 下载量·评论数·月活·估计用户数(能找到的) |
| 融资 | 轮次·金额(若有) |
| 差评痛点 | 从用户评论/App Store 差评/Steam 差评中提炼 2–3 条 |

每条数据**必须记录来源 URL**;若某条无法从当前页找到,标 `[未核实]`,不猜测。

工具失败处理见「错误处理」节。

### Phase 3 — 信息分级

对 Phase 2 收集的每条信息,按以下规则打标签:

| 标签 | 条件 |
|---|---|
| `[已核实]` | 有明确来源 URL + 日期可追溯 |
| `[未核实]` | 无来源链接,或链接已失效 |
| `[AI推测]` | 基于已知信息推断,非直接来源 |
| `[存疑]` | 两个来源数据矛盾,无法确定哪条正确 |

**评估门只采信 `[已核实]` 数据。**`[AI推测]` 和 `[未核实]` 仅供参考,不得在报告中当事实陈述。

### Phase 4 — 写调研报告 + 回填

1. 写 `调研/idea-NNNN_调研.md`(结构见下方「调研报告结构」)。
2. 回填 idea 文件 frontmatter:
   ```yaml
   research_ref: 调研/idea-NNNN_调研.md
   status: researched
   updated: {今天日期}
   ```
3. 更新 `想法池/_状态总览.md` 对应行的 `status` 为 `researched`、`updated` 为今天日期。

**若 Phase 2 遇到 web 工具失败导致调研不完整,status 不推进(见「错误处理」)。**

### Phase 5 — 给评估留钩子

在报告 `## 市场空白与差异化机会` 节之后,添加一段「评估门参考」,归纳:

- **价格带**:市场主流定价区间(仅含 `[已核实]` 数据)。
- **用户抱怨最多的痛点**:从差评提炼的前 2–3 条,供评估门挑战核心假设。
- **市场空白点**:现有竞品缺失的功能/场景/人群(若搜不到有力证据,标 `[AI推测]`,不当事实写)。

收尾回执:

```
调研完成:idea-NNNN「{标题}」→ researched。
找到竞品 {N} 个,已核实事实 {M} 条。
下一步:/评估 idea-NNNN
```

## 调研报告结构

写入 `调研/idea-NNNN_调研.md`:

```markdown
---
idea: idea-NNNN
date: YYYY-MM-DD
platform: {platform}
sources_count: {已核实来源数}
---

# idea-NNNN 调研报告

## 竞品清单

| 竞品 | 做什么 | 定价 | 体量 | 来源 |
|---|---|---|---|---|
| {名称} | {功能} | {价格} [已核实] | {规模} [已核实] | [链接](URL) |

## 价格带

{市场主流定价区间描述,每条标 [已核实]/[未核实]}

## 用户痛点(从差评提炼)

- {痛点 1} [已核实] — 来源:{链接}
- {痛点 2} [已核实] — 来源:{链接}

## 市场空白与差异化机会

{现有竞品缺失的场景/人群/功能;无有力证据的推断标 [AI推测]}

## 信息可信度说明

- 已核实({N} 条):有来源链接的事实,评估门可采信。
- 未核实({M} 条):{列出哪些信息无法找到来源}
- 存疑({K} 条):{列出矛盾信息及各自来源}
```

---

## 防幻觉硬约束

1. **无来源链接的竞品事实,一律标 `[未核实]`,绝不当事实写进报告。**
2. **数字(定价/体量/融资)必须带来源 URL + 标采集日期**(数字会过期,日期是可信度的一部分)。
3. **模型记忆不是来源**:Claude 训练数据里"知道"某竞品的信息,不等于有来源链接,必须用 `web_reader` 实际拉取后才可标 `[已核实]`。
4. 评估门只采信 `[已核实]` 数据——调研报告的价值上限由已核实条数决定。

---

## 错误处理

### 搜不到直接竞品

不是失败,是强信号。**不默认当蓝海。**

处理方式:
- 在 `## 竞品清单` 写:「未找到直接竞品(关键词:{词},主源:{源},搜索日期:{日期})」。
- 在 `## 市场空白与差异化机会` 注明:「无直接竞品可能意味着需求尚未被验证,亦可能意味着需求本身不存在——交评估门判断,不在此下结论。」
- `status` 正常推进到 `researched`(调研已完成,结果就是无直接竞品)。

### web 工具超时 / 失败

- 单个工具调用失败 → 换关键词重试一次;仍失败 → 在该竞品行标 `[调研失败:工具超时]`。
- 若多数竞品数据均无法获取,报告顶部标「**调研未完成**」,并注明失败范围。
- **`status` 不推进到 `researched`**;在 idea frontmatter 的 `updated` 字段写今日日期,保持 `status: refined`。
- 提示用户:「web 工具异常,调研未完成。请稍后重试 `/调研 idea-NNNN`。**绝不用模型记忆冒充联网结果。**」

### 信息零散 / 矛盾

- 两条来源数据不一致 → **两条都列出**,各附来源 URL,统一标 `[存疑]`。
- **不强行调和**,不取平均,不选"感觉更可信"的一条。
- 在 `## 信息可信度说明` 注明存疑条目,交评估门定夺。
