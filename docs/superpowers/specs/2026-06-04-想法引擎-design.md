# 想法引擎(Spark)设计文档

> 状态:设计已确认(2026-06-04),待转写实现计划
> 形态:本地文件知识工程 + Claude Code / Codex Skill 套件
> 灵感来源:dontbesilent2025/dbskill 的「原子库 + skill 解耦」与「decision 决策系统」模式

---

## 1. 目标与定位

把平时**破碎的想法**(常常只是一句话)沉淀进一个本地知识工程,由 AI 协助:

0. **补背景**:一句话里带不懂的术语/项目背景,先标记、确认后查清并沉淀(让 AI 听懂你的简写);
1. **一起完善**粗糙想法(交互式对话,像 dbskill);
2. **联网调研**真实竞品(带来源链接,不脑补);
3. 给出**可行性评估**(go / no-go / maybe 决策门);
4. 为通过的候选产出**商业设计 + PRD 等设计文档**。

**核心差异化:** 想法不是流水线里的一次性项目,而是池子里的**原子**。单个想法评估失败不删除,而是**降级回池、带上"死因"**,成为未来**组合(找火花)**的燃料。"多个想法拼合更容易出火花"是本系统的灵魂。

### 1.1 非目标(YAGNI)

- 不做独立 Web/App 产品,不做前后端、数据库、部署。
- 不做 dbskill 式的 save/restore/report 存档三件套(本系统状态天然落在文件里)。
- 不给提示词系统上重型测试框架(只做轻量黄金样例回归)。

## 2. 关键设计决定(已与用户确认)

| # | 决定 | 理由 |
|---|---|---|
| D1 | 形态 = 本地文件知识工程 + Skill 混合 | 想法长期沉淀、可演化、可回填 |
| D2 | 调研 = 联网实时(web_search + web_reader MCP),每条带来源 | 竞品信息必须可核查,杜绝编造 |
| D3 | 流程**解耦**,非强制线性流水线 | 各原子/候选各有自己的阶段状态 |
| D4 | 「找火花」是**横切能力**,挂在「完善②」和「独立扫描⑤」两个触发点 | 养肥想法的当下最易照亮邻近想法 |
| D5 | 评估失败 = **降级回池 + 写死因**,不删除 | 死因是找火花做互补匹配的结构化燃料 |
| D6 | 一个想法 = **一个 markdown 文件**(非 JSONL) | 想法是活的、会长肉、要互链,近似 dbskill content-system 的"内容单元" |
| D7 | **活文件 / 死文件分离**:想法池可变,调研·评估·产出为写完不改的快照 | 永远能回看"当时基于什么调研做了什么判断" |
| D8 | 数据结构 day 1 钉死,skill 分两期建 | 改提示词廉价,迁移已存数据昂贵 |
| D9 | 防幻觉**硬约束**:无来源链接的竞品事实一律标 `[未核实]`,绝不脑补冒充联网结果 | 系统可信度生死线 |
| D10 | 新增横切**背景/知识层**:遇不懂的术语/项目,**标记+确认后**再补背景并沉淀 | 让 AI 听懂你的简写,每次对话快速抓核心 |

## 3. 架构:流水线 + 想法生命周期

「找火花」横切 ②⑤,流程解耦、可中途停。

```text
想法池(原子,永不删除)
  │
  │  ① 捕捉:一句话也行,先进池子(原文逐字保留)
  ▼
② 完善 ⇄ 找火花   ← 交互式对话(一次问一个),养成可调研的想法
  │                 收尾顺手扫池子:「这个和 idea-0031 可能互补」
  │   ⤷ 太空泛/伪概念 → 当场降级回池,不浪费调研
  ▼
③ 调研:联网拉真实竞品(产品/定价/体量/融资/差评痛点),每条带来源
  ▼
④ 评估门:逐条挑战核心假设 → verdict
  │      go    → 升为候选
  │      maybe → 回调研补 / 回完善改方向
  │      no-go → 降级回池(写死因)──┐
  ▼                                  │
⑤ 找火花(也可独立全池扫描)←————————┘
     扫所有原子(含死的)提组合 → 新建 combo 候选 → 回 ② 重新养
  │
  ▼  候选(单原子或组合)通过评估门后
⑥ 商业设计(定位/模式/定价/GTM)
  ▼
⑦ PRD 及设计文档(架构/技术方案/task list)
```

死亡重组闭环:`no-go 死因` → 找火花按死因互补匹配(如"缺流量入口" × "缺变现路径")→ combo 候选 → 回完善 → 再调研评估。

**横切背景层:** 「背景调查」与「找火花」一样是横切能力,不在主线上。完善(及任意阶段)遇到不确定的术语/项目背景时,**标记并征得你同意后**联网查(公开术语)或问你(私有项目),沉淀进 `背景库/`;所有 skill 开头读它,确保对话快速抓核心。

## 4. 数据模型

### 4.1 目录结构

```text
{工程根}/
├── 想法池/
│   ├── _状态总览.md                 # 第一入口:一张表看全部想法在哪个阶段
│   ├── idea-0001_一句话标题.md       # 活的原子(可变,不断长肉);死的也留这,只是状态变
│   └── ...
├── 调研/
│   └── idea-0001_调研.md            # 调研快照(写完基本不改,带来源链接)
├── 决策档案/
│   └── idea-0001_评估_20260604.md   # 评估快照(写完不改)
├── 产出/
│   ├── idea-0007_商业设计.md         # 候选 = status:candidate 的 idea 原子,沿用 idea-NNNN,无独立命名空间
│   └── idea-0007_PRD.md
├── 背景库/                          # 横切知识层,所有 skill 开头读
│   ├── 概念词典.md                  # 公开术语,实时生长(借 dbskill 高频概念词典)
│   ├── 项目档案/                    # 私有项目背景,每项目一文件 [本人]
│   │   └── proj-001_名字.md
│   └── _背景索引.md                 # 术语/项目 → 文件 + 一句话,供快速读取
└── 规则/
    ├── schema与阶段定义.md
    ├── 品类与平台映射.md            # 游戏→Steam, App→iOS … 可配置
    ├── 死因词表.md                  # 受控词表,供找火花匹配
    └── eval样例.md                  # 黄金样例回归
```

### 4.2 想法原子 schema(frontmatter)

```yaml
---
id: idea-0042
title: 一句话标题
status: raw            # raw→refining→refined→researching→researched→evaluated→candidate→documented
kind: single           # single | combo
parents: []            # combo 时填来源原子,如 [idea-0007, idea-0031];原子永不被删
tags: [AI, 效率工具]    # 主题标签 → 找火花按主题聚类
category: 游戏          # 品类:游戏|App|SaaS|内容|硬件|工具|…(可扩展)
platform: Steam        # 首发平台:Steam|iOS|Android|Web|小程序|抖音|公众号|…(完善时按映射表自动填,可覆盖)
effort: null           # 工作量预估 S/M/L/XL  [AI推测→修正]
excitement: null       # 你的兴奋度 1-5  [本人]
verdict: null          # null | go | no-go | maybe
death_reason: null     # no-go 时必填,取自死因词表
created: 2026-06-04
updated: 2026-06-04
research_ref: null      # → 调研/idea-0042_调研.md
eval_ref: null          # → 决策档案/idea-0042_评估_YYYYMMDD.md
artifacts: []           # → 产出/ 下商业设计、PRD
relationships:          # 与池中其他原子的关系(供找火花)
  - { type: 互补, target: idea-0031 }   # 互补|撞车|同源|反例
---
```

### 4.3 想法原子 body(活文件,完善时长肉)

```markdown
## 原文 [本人]
> {你最初丢进来的那句话,逐字保留,永不改写}

## 完善后描述
{解决谁的什么痛点 · 初步产品形态}

## 核心假设
- [假设] {这想法成立依赖的关键假设 —— 评估门逐条挑战}

## 待调研
- 竞品关键词:{...}
- 要核实的事实:{...}

## 完善记录 [AI 元记录]
{关键转折、你否决的方向、为什么降级}
```

**来源标签**(借 dbskill 决策系统):`[本人]` / `[AI推测]` / `[AI结论]` / `[修正]`。

### 4.4 状态总览(`_状态总览.md`)

一张表(id / 标题 / status / verdict / category / platform / excitement / effort / updated),每个 skill 收尾时更新自己那一行。可按 excitement 降序 / effort 升序排,一眼看到该先动哪个。**它是你的第一入口。**

## 5. Skill 套件

设计原则:每个 skill 小而专,只读写自己那几个文件,收尾推荐下一步(沿用 dbskill skill 互推)。

### 5.1 第一期 —— 核心环路(7 个)

| skill | 触发 | 职责 | 读 | 写 |
|---|---|---|---|---|
| 主入口 | `/想法` `/idea` | 只路由,不干活 | `_状态总览` | — |
| 捕捉 | `/捕捉 "一句话"` | 零摩擦建 raw 原子 | — | `想法池/idea-NNNN.md` |
| 完善 | `/完善 idea-NNNN` | 交互式养肥;降级伪概念;收尾找火花 | 全 `想法池/` + 映射表 + `背景库/` | 改写当前 idea 文件 |
| 背景调查 | `/背景`(或被完善/捕捉调用) | 标记不懂的术语/项目→确认后:公开术语联网查、私有背景问你→沉淀 | 当前对话 + `背景库/` | `背景库/`(概念词典/项目档案/索引) |
| 调研 | `/调研 idea-NNNN` | 按 category/platform 联网扒竞品,带来源 | idea + 映射表 | `调研/…`,回填 research_ref |
| 评估 | `/评估 idea-NNNN` | 决策门,挑战假设出 verdict;no-go 降级 | idea + 调研报告 | `决策档案/…`,回填 verdict/eval_ref |
| 找火花 | `/找火花`(独立全池扫描)或被完善调用(轻量,只扫与当前想法相关) | 扫全池(含死原子)提组合,建 combo 候选 | 全 `想法池/` | 新建 `kind=combo` 原子 |

### 5.2 第二期 —— 产出套件(2 个,等真有候选过门再建)

| skill | 触发 | 职责 | 读 | 写 |
|---|---|---|---|---|
| 商业设计 | `/商业 idea-NNNN` | 给候选出定位/模式/定价/GTM | 候选 + 调研 + 评估 | `产出/…商业设计.md` |
| PRD | `/prd idea-NNNN` | 出 PRD + 架构/技术方案/task list | idea + 商业设计 | `产出/…PRD.md` 等 |

### 5.3 互推关系

```text
捕捉 ─▶ 完善 ─▶ (找火花 ⇄) ─▶ 调研 ─▶ 评估
                                       │ go ─▶ 商业设计 ─▶ PRD
                                       │ no-go ─▶ 降级回池(死因)─▶ 找火花
```

## 6. 两个交互环节的内部流程

### 6.1 ② 完善

```text
Phase 0  读 raw 原子 + 扫池子 tags(为收尾找火花预热)
Phase 1  一句话复述 → 问:「这解决谁的什么痛?」
Phase 2  语言陷阱检测(借 diagnosis 第一层):核心词模糊就逼定义
Phase 3  产品形态 + 按映射表填 category→platform,问你确认/覆盖
Phase 4  抽 1-3 条「核心假设」← 留给评估门的靶子
Phase 5  问 excitement([本人]) + AI 初估 effort
Phase 6  列「待调研」:竞品关键词 + 要核实的事实
Phase 7  收尾:调用找火花扫池 → 更新文件 + 状态总览
```

**出闸条件(够肥才放行到调研):** ①能一句话说清"为谁解决什么" ②category+platform 已定 ③≥1 条核心假设 ④≥2 个竞品关键词。

**降级判据(不进调研,留池当燃料):** 核心词无法定义 `#伪概念` / 说不出痛点 `#太空泛` / 本质是情绪愿望非产品 `#不是产品`。

### 6.2 ④ 评估门

```text
Phase 0  读 idea + 调研报告。无 research_ref 直接拦:「先 /调研」
Phase 1  逐条挑战核心假设,标 成立/存疑/证伪,必须引调研证据([已核实])
Phase 2  竞品现实核对:有人已做得很好?差异化在哪?(撞车检测)
Phase 3  商业快筛(借 diagnosis 公理):真实付费意愿?effort vs 资源?平台现实?
Phase 4  出 verdict:
         go    → 假设多数成立 + 有差异化 + effort 可承受 → 候选
         maybe → 关键假设缺数据 → 回调研补 / 回完善改方向
         no-go → 假设被证伪 / 撞车无差异 / effort 过大 → 降级回池 + 【必写死因】
Phase 5  写评估快照(写完不改,带时间戳)+ 回填 + 更新状态总览
```

**死因 = 找火花燃料(核心机制):** no-go 必须写受控词表里的死因。初版词表:`缺流量入口` / `缺变现路径` / `缺技术能力` / `撞车无差异` / `需求太弱` / `effort过大` / `平台限制`(放 `规则/死因词表.md` 可扩展)。找火花按死因互补匹配。

### 6.3 背景调查(横切,标记+确认触发)

```text
Phase 0  读 背景库/_背景索引(已沉淀的不重复查)
Phase 1  识别当前对话里"不确定/未知 名词 + 项目背景引用",分类:公开术语 / 私有背景
Phase 2  标记 + 征求同意:「这几个我不确定:A(公开,联网查?)、B(像你的私有项目,讲两句?)」等你点头
Phase 3a 公开术语(同意后):web_search 定义 + 关键事实,带来源 → 写 概念词典.md([已核实]+日期)
Phase 3b 私有背景(同意后):问 3 点(是什么/现状/为什么和当前想法相关)→ 写 项目档案/proj-xxx.md([本人])
Phase 4  更新 _背景索引 → 回原对话带着补全的背景继续
```

**捕捉零摩擦原则:** 捕捉阶段不打断,若检测到未知词只在原子里记 `待补背景:[词]`,留到完善或手动 `/背景` 再处理。

**所有 skill 开头读 `背景库/_背景索引`**,这是"快速抓到核心"的落地点;公开术语词典随对话实时生长,是 dbskill「高频概念词典」的活体版。

## 7. ③ 调研数据流 + 防幻觉

```text
Phase 0  读 idea(待调研关键词 + category + platform)+ 映射表 → 决定扒哪个市场
Phase 1  竞品发现(web_search):
         游戏/Steam → Steam 同类标签榜、SteamDB
         App/iOS    → App Store 榜单 + 类似 app
         SaaS/Web   → 通搜 + Product Hunt
         内容/抖音/公众号 → 平台搜索 + 对标账号
Phase 2  竞品深挖(web_reader 逐个):做什么/定价/体量/融资/差评痛点
Phase 3  信息分级:每条打 [已核实+链接] / [AI推测] / [未核实]
Phase 4  写 调研/idea-NNNN_调研.md,回填 research_ref,status→researched
Phase 5  给评估留钩子:价格带、用户抱怨最多的痛点、市场空白点
```

调研报告结构:竞品清单表(竞品/做什么/定价/体量/**来源**)→ 价格带 → 用户痛点(从差评提炼)→ 市场空白与差异化机会 → 信息可信度说明。

### 7.1 防幻觉硬约束(D9)

- **没有来源链接的竞品事实,一律标 `[未核实]`,不准当事实写。** 评估门只采信 `[已核实]`。
- 数字(定价/体量/融资)必须带来源 + 标日期(会过期)。

### 7.2 错误处理

| 情况 | 处理 |
|---|---|
| 搜不到直接竞品 | 不是失败,是强信号。写"未找到直接竞品",交评估门判(借 diagnosis:没竞品常意味没需求,不默认当蓝海) |
| web 工具超时/失败 | 报告标"调研未完成",status **不**推进,提示重试。**绝不脑补冒充联网结果** |
| 信息零散/矛盾 | 两条都列 + 各自来源,标 `[存疑]`,不强行调和 |

## 8. 测试:轻量黄金样例回归

不上重型测试框架。`规则/eval样例.md` 存一组输入 + 期望行为,每次改 skill 手动跑一遍:

| 输入样例 | 期望行为 |
|---|---|
| 伪概念:"我想做个 AI 改变世界" | 完善判 `#太空泛`,降级,不进调研 |
| 清晰想法:"给 Steam 独立游戏做差评聚合工具" | 完善出闸 → 调研扒到竞品 → 评估出 verdict |
| 明显撞车的想法 | 评估识别撞车,死因 `撞车无差异`,no-go |
| 两个互补死原子(缺流量 + 缺变现) | 找火花配成 combo 候选 |
| 一句话带未知术语(如"做个 roguelike 的 SteamDB") | 背景调查标记 SteamDB/roguelike,确认后联网查并写进概念词典 |

对应 `ai-regression-testing` 思路,保持轻量。

## 9. 实施分期

- **第一期(核心飞轮):** 数据结构(§4 全部,含 `背景库/`)+ 主入口 / 捕捉 / 完善 / **背景调查** / 调研 / 评估 / 找火花(7 个)+ 三张规则表(品类平台映射 / 死因词表 / eval 样例)。目标:跑通"想法进来 → 补背景 → 完善 → 调研 → 评估 → 死了重组"。
- **第二期(产出套件):** 商业设计 / PRD,等第一期真有候选通过评估门再建。
